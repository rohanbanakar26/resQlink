# ResQLink Visual Design System

## Brand Direction
ResQLink should feel like a premium civic-tech platform built for urgent community coordination.

Core emotional signals:
- trust
- speed
- seriousness
- clarity
- visible real-time action

## Visual Tokens

### Colors
- Background: `#F6F8F4`
- Panel background: `rgba(255,255,255,0.92)`
- Primary trust green: `#145A41`
- Secondary green: `#2E8B66`
- Emergency red: `#D93C2F`
- Warning amber: `#E9A23B`
- Text: `#14201B`
- Soft muted text: `#7A8D84`

### Typography
- Headings: `Fraunces`
- UI/body: `Space Grotesk`

Rules:
- heading copy should be bold and short
- body copy should be compact and readable
- avoid dense paragraphs on operational screens

### Shape
- Card radius: `20px - 28px`
- Buttons: pill or soft rounded rectangles
- Borders: subtle, low-contrast
- Shadows: soft and wide, not sharp

## Page-by-Page System

### 1. Landing Page
Purpose:
- explain the product in one glance
- create trust quickly
- drive users to report, register, or explore

Structure:
1. Hero
2. Live stats
3. Five problem domains
4. Role cards
5. Five pillars

Design notes:
- strongest heading in the whole app
- one dominant emergency CTA
- supportive NGO CTA secondary
- cards should feel premium and calm

### 2. App Home
Purpose:
- be the public-facing mobile command surface

Structure:
1. Active alert strip
2. Greeting + location
3. Emergency CTA panel
4. Problem category list
5. Nearby NGOs carousel

Design notes:
- one primary action should always be visible
- category cards should feel instantly tappable
- nearby NGO strip should feel trustworthy and human

### 3. Domain Pages
Purpose:
- combine campaign proof + request intake

Structure:
1. Category hero
2. Campaign cards
3. Smart matching explainer
4. Request form toggle
5. Confirmation state

Design notes:
- keep the request form secondary until the user taps
- use campaign proof to create trust before submission

### 4. NGO Dashboard
Purpose:
- feel like a command center

Structure:
1. KPI strip
2. Top critical areas
3. Suggested actions
4. Request list
5. Heatmap and trend views

Design notes:
- overview tab should be intelligence-first
- requests tab should be operational
- heatmap tab should feel spatial and strategic

### 5. Volunteer Dashboard
Purpose:
- make the volunteer feel active, useful, and mobile

Structure:
1. Queue header
2. Status / quick stats
3. Assigned task cards
4. Navigation and completion actions

Design notes:
- task cards should prioritize urgency, distance, and action buttons

### 6. Search
Purpose:
- discovery of NGOs and volunteers

Structure:
1. Sticky search area
2. NGO / volunteer tabs
3. Result cards

Design notes:
- cards should emphasize trust score, skills/services, and location

### 7. News
Purpose:
- show proof of impact

Structure:
1. Page intro
2. Platform impact banner
3. NGO update cards

Design notes:
- should feel editorial but still operational

### 8. Profiles
Purpose:
- establish trust and social proof

Structure:
1. Cover
2. Identity block
3. Metrics
4. Skills / services
5. Action buttons

Design notes:
- verification and impact should be visible above the fold

### 9. About
Purpose:
- explain mission, trust, and legitimacy

Structure:
1. App identity
2. Live stats
3. Feature cards
4. Trust section
5. Contact

## Motion Rules
- subtle fade-in on major sections
- soft hover lift on tappable cards
- pulse only for emergency/live indicators
- no heavy animation on operational dashboards

## UX Rules
- one clear CTA per major section
- never overload a card with more than one main decision
- always show status visually before textually
- maps should support decisions, not dominate everything
